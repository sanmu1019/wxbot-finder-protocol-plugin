# 视频下载与解密

## 下载输入

插件优先读取：

```text
media.full_url
```

没有 `full_url` 时拼接：

```text
media.url + media.url_token
```

URL 必须：

- 使用 HTTP 或 HTTPS。
- 主机属于已知 Finder CDN。
- 路径为视频下载路径。
- 包含 `encfilekey`。
- 包含 `token`。

## 为什么需要 decodeKey

部分 Finder 视频响应虽然：

```text
HTTP 200
Content-Type: video/mp4
```

但下载文件头不是标准 MP4。视频前部被 Finder 流加密，必须使用同次详情响应中的
`decode_key`。

## 解密范围

插件读取文件前：

```text
131072 字节
```

即 128 KB，然后使用 ISAAC64 伪随机流做 XOR。

解密后正常 MP4 文件头通常能看到：

```text
ftyp
```

## 为什么只处理前 128 KB

当前 Finder 视频加密主要影响文件头和前部数据。协议实现中已经验证对前 128 KB
解密后可以恢复可识别视频文件。

不要对整个大文件重复解密，否则会破坏未加密部分。

## 文件检查

插件接受以下文件头：

```text
MP4: 头部包含 ftyp
WebM/Matroska: 1A 45 DF A3
AVI: RIFF
FLV: FLV
```

文件小于 1 KB 会被拒绝。

## 发送顺序

```text
send_video_msg
-> 失败
-> send_file_msg
```

无论发送成功还是失败，插件都会删除下载临时文件。

## Key 配对要求

以下数据必须来自同一次协议响应：

```text
full_url
url_token
decode_key
```

混用不同请求、不同卡片或不同时间的 URL 和 Key，会导致：

- 下载失败。
- token 过期。
- 解密后仍不是视频。
- 文件损坏。

## 测试

单元测试验证 ISAAC64 XOR 的可逆性：

```text
原数据
-> 使用同一 key XOR
-> 再次使用同一 key XOR
-> 恢复原数据
```

真实媒体测试仍需已授权的 bot 登录态和视频号卡片。
