# 协议直刷原理

## 核心结论

协议直刷不生成 `sph`，也不调用元宝。

只要卡片 XML 中包含同一张视频的：

```text
objectId
objectNonceId
```

就可以使用 bot 当前 protocol-core 登录态刷新 Finder 详情。

## 完整链路

```text
微信用户引用视频号卡片
-> wxbot 收到消息事件
-> 插件读取引用卡片 XML
-> 提取 objectId/objectNonceId
-> POST /Finder/GetCommentDetail
-> protocol-core 读取当前 bot 登录态
-> 调微信 findergetcommentdetail
-> 返回媒体 URL、url_token、decode_key
-> 插件下载视频
-> 必要时解密前 128 KB
-> send_video_msg
-> 失败时 send_file_msg
```

## bot 协议登录态的作用

protocol-core 通过 bot 的 wxid 查找：

```text
MMTLS 地址和 Key
Cookie
SessionKey
ClientSessionKey
设备信息
RSA 公钥
```

然后构造：

```text
wechat.FinderGetCommentDetailRequest
```

调用：

```text
/cgi-bin/micromsg-bin/findergetcommentdetail
```

默认 CGI：

```text
3906
```

因此协议登录态失效时，插件无法刷新媒体。

## 请求字段

```text
objectId
objectNonceId
wxid
cgi
```

`objectId` 与 `objectNonceId` 必须来自同一张卡片。旧 nonce、错误配对或截断字段
会导致接口失败或返回空对象。

## 媒体字段

protocol-core 从 Finder 响应中读取：

```text
Url
ThumbUrl
MediaType
VideoPlayLen
Width
Height
Md5Sum
FileSize
Bitrate
```

并从 protobuf unknown fields 中补取：

```text
field 12 -> decode_key
field 13 -> url_token
```

最终返回：

```text
full_url = url + url_token
decode_key
```

这两个值必须来自同一次 Finder 响应。

## 为什么不使用卡片旧 URL

原始卡片可能包含 Finder CDN 地址，但经常存在：

- 缺少 token。
- token 已过期。
- 缺少 decodeKey。
- 下载内容仍然加密。

协议直刷会用当前登录态重新请求详情，获得同批次的新 URL 和 Key，比直接使用
旧卡片 URL 稳定。

## 与短链方案的关系

协议直刷：

```text
卡片 -> 媒体
```

短链方案：

```text
卡片 -> sph -> 元宝 -> 媒体
```

协议直刷适合机器人直接下载发送。它不会产生可复制分享的
`https://weixin.qq.com/sph/...`。

## 并发和风险

Finder CGI 是微信私有协议能力。建议：

- 使用明确触发词。
- 不批量解析历史卡片。
- 不枚举 object ID。
- 限制群聊调用频率。
- 不进行压力测试。
